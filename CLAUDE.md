# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

This repo authors and publishes [dev container Features](https://containers.dev/implementors/features/) to GHCR following the [Features distribution spec](https://containers.dev/implementors/features-distribution/). It currently ships a single Feature: `claude-code` â€” a Claude Code installer + host-credentials bridge for dev containers.

## Common commands

All testing uses the `@devcontainers/cli` (install with `npm install -g @devcontainers/cli`). Commands are run from the repo root.

- **Autogenerated test** (defaults only): `devcontainer features test --skip-scenarios -f claude-code -i <baseImage> .`
- **Scenario tests** (from `test/claude-code/scenarios.json`): `devcontainer features test -f claude-code --skip-autogenerated --skip-duplicated .`

There is no build or lint step. Validation of `devcontainer-feature.json` runs in CI via `devcontainers/action@v1` with `validate-only: "true"` (see `.github/workflows/validate.yml`).

Publishing to GHCR is **manual** via the `workflow_dispatch`-only `release.yaml` workflow; it also opens an automated PR with a regenerated `src/claude-code/README.md` (canonical source of truth). Locally, `src/<feature>/README.md` is regenerated by a project-scoped PostToolUse hook (see "Project-scoped Claude Code hooks" below) â€” do not hand-edit it.

## Architecture

### The `claude-code` Feature (`src/claude-code/`)

Unlike a typical "single shell command in /usr/local/bin" Feature, this one uses **all three** devcontainer lifecycle hooks plus a build-time install:

- `install.sh` (build-time, **root**): downloads the Claude Code bootstrap binary from `downloads.claude.ai`, verifies SHA-256 against `manifest.json`, caches it at `/opt/claude-code/cache/claude`, installs jq/curl/sudo if missing, and stages the lifecycle scripts + `_lib.sh` under `/usr/local/share/claude-code/`. **Does not** run `claude install` (that would land in root's home).
- `onCreate.sh` (post-create, **once**): runs `claude install <channel>` as the target user. This is the prebuild-cacheable step â€” it touches the user's home but **not** the host-bind mount, so it works inside Codespaces prebuilds.
- `postCreate.sh` (post-create, **once**, after workspace mount): forwards host OAuth credentials + minimal `.claude.json` from the read-only host mount (`/host_claude/...`) into the target user's `~/.claude/`. Then optionally adds plugin marketplaces (`CLAUDE_MARKETPLACES`) and installs plugins (`CLAUDE_PLUGINS`).
- `postStart.sh` (every container start): refreshes credentials if the host's `expiresAt` is strictly newer, merges account fields on userID change (without clobbering trust/remote-control), and idempotently patches `.claude.json` (workspace trust + `remoteControlAtStartup`) and `settings.json` (`permissions.defaultMode`). Optionally spawns `claude remote-control --spawn worktree` as a backgrounded daemon (requires the workspace to be a git repo).
- `_lib.sh` (sourced, **not** executed): shared helpers â€” target-user auto-detection (lowest UID â‰Ą 1000 with a real login shell), `resolve_target_paths`, `run_as_target` (direct / `su -` / passwordless `sudo`), `install_for_target` (atomic temp-then-mv), `run_as_target_background` (nohup daemon spawn).

### Critical conventions in this Feature

- **Target user**: never assumed to be `vscode`. Either `CLAUDE_TARGET_USER` option or auto-detection picks it. Always go through `resolve_target_paths` + `run_as_target` â€” do not hardcode usernames or `$HOME`.
- **Host mount**: `mounts` in the manifest bind-mounts `$HOME/.claude.json` and `$HOME/.claude/` from the host into `/host_claude/`. The Feature schema does not allow a `readonly` flag on Feature mounts â€” the lifecycle scripts treat the path as read-only by convention (only read, never write). See `src/claude-code/NOTES.md` for the schema limitation. Missing host files â†’ soft-skip (`exit 0` with a log), not failure.
- **CSV options**: `marketplaces` and `plugins` are comma-separated strings (devcontainer Features have no native array option type). The parser trims whitespace and skips empty items.
- **Idempotency**: `postStart.sh` runs on every container start, so its JSON patches diff against the current file and only write when something actually changed.
- **`set -euo pipefail`** is the convention in every lifecycle script. `_lib.sh` deliberately does **not** set it (caller's responsibility).
- **Logging**: every script sets `SCRIPT_TAG="..."` before sourcing `_lib.sh`; use `log`/`warn`/`fail` from the lib for tagged output.

### Project-scoped Claude Code hooks (`.claude/hooks/`)

Two hooks are wired in `.claude/settings.json` so the generated Feature READMEs stay in sync without manual steps:

- `block-feature-readme-edit.sh` (PreToolUse, matcher `Write|Edit|MultiEdit|NotebookEdit`): denies edits to any `src/<feature>/README.md`. Reason surfaced to the model: edit `devcontainer-feature.json` or `NOTES.md` instead.
- `regenerate-feature-readme.sh` (PostToolUse, matcher `Write|Edit|MultiEdit`): after edits to `src/<feature>/devcontainer-feature.json` or `src/<feature>/NOTES.md`, runs `devcontainer features generate-docs --project-folder . --namespace <owner-from-git-remote>/<repo>`. Same generator the release workflow invokes via `devcontainers/action@v1`, so local output matches the release-PR output. Soft-skips when the `devcontainer` CLI or `git remote.origin.url` is missing.

### Test layout (`test/claude-code/`)

The devcontainer CLI maps test files by name:
- `test.sh` â€” autogenerated default test (defaults only, no scenarios).
- `scenarios.json` keys (`bypass-permissions`, `no-extras`, `auto-detect-user`, `with-marketplaces`) â€” each scenario's assertion script must be `<scenarioName>.sh` in the same directory.

All scenarios target `mcr.microsoft.com/devcontainers/base:ubuntu` (or `:debian` for `auto-detect-user`) with `remoteUser: vscode`. Scripts `source dev-container-features-test-lib` and use `check`/`reportResults`. CI exercises the credential-forwarding path against minimal stub `~/.claude.json` + `~/.claude/.credentials.json` files that the workflow writes on the runner before invoking `devcontainer features test` (see `.github/workflows/test.yaml`). Plugin/marketplace installs are still not reachable since the stub credentials don't authenticate to Anthropic's API.

### Adding a new Feature
1. Create `src/<name>/devcontainer-feature.json` and `src/<name>/install.sh`.
2. Add `test/<name>/test.sh` (autogenerated) and optionally `test/<name>/scenarios.json` + per-scenario scripts.
3. Add the Feature name to **both** matrix lists in `.github/workflows/test.yaml` (`test-autogenerated` and `test-scenarios`).
4. Bump `version` in the manifest on every change that should publish a new tag.

## Notes from README

- Default published namespace is `ghcr.io/<owner>/<repo>/<feature>`; the release workflow also publishes a collection-level metadata package at `ghcr.io/<owner>/<repo>`.
- For the release workflow's auto-PR to work, the repo needs *Allow GitHub Actions to create and approve pull requests* enabled in **Settings â†’ Actions â†’ General â†’ Workflow permissions**.
- Features publish as **private** by default in GHCR â€” flip them to public per-package to stay on the free tier.
